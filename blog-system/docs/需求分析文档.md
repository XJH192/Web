## 1. 项目背景

随着个人内容创作、技术分享和线上社交的发展，传统静态博客只具备单向发布能力，难以满足多用户协作、互动通知、内容安全审核和智能写作等需求。本项目在 Hexo Ciallo～(∠・ω< )⌒☆ 主题基础上，建设一个前后端分离的多用户智能博客系统。

系统以普通用户和管理员为核心角色。普通用户可以注册登录、发布文章、管理文章、评论回复、点赞、关注、私信、维护个人相册，并使用 AI 完成摘要、大纲、标签、分类和博客问答。管理员负责用户、文章、评论、分类、标签及 AI 日志管理。

为兼顾发布效率和内容安全，文章与评论采用“AI 先审、人工兜底”的流程：正常内容由系统直接公开，疑似广告、辱骂、低俗、诈骗等内容转交管理员判断。这样既减少管理员重复审核工作，又避免风险内容直接公开。

## 2. 建设目标

1. 建设一个可运行、可演示、可扩展的多用户博客平台。
2. 实现用户、文章、评论、社交、相册、私信和通知等完整业务闭环。
3. 使用 MySQL 持久化正式业务数据，并使用浏览器本地存储保存未发布草稿。
4. 接入 DeepSeek 兼容接口，提高写作效率和内容检索能力。
5. 通过本地规则与管理员复核保障内容安全。
6. 提供清晰的角色权限、隐私保护和数据级联删除机制。

## 3. 用户角色分析

### 3.1 游客

游客是尚未登录系统的访问者。

主要权限：

- 浏览站点公开页面和主题内容。
- 进入登录、注册页面。
- 不允许调用 AI、发布文章、评论、点赞、关注、私信或管理相册。

### 3.2 普通用户

普通用户是系统主要内容生产者和互动参与者。

主要权限：

- 注册、登录和查看个人资料。
- 发布、编辑、删除自己的文章，保存草稿。
- 使用分类、标签和附件。
- 浏览公开文章、点赞、评论、回复和评论点赞。
- 搜索用户和文章。
- 关注、取关其他用户，查看粉丝和关注列表。
- 与其他用户私信。
- 管理自己的 8 张个人相册图片。
- 使用 AI 写作辅助和博客问答。
- 查看互动通知并标记已读。

限制：

- 不能修改或删除其他用户的文章、评论和相册。
- 非互关用户最多向同一用户发送 3 条私信。
- 风险文章和风险评论需要管理员复核。

### 3.3 管理员

管理员负责平台治理，同时拥有一个可独立维护的个人账号。

主要权限：

- 查看系统统计数据。
- 查看用户列表，修改角色、封禁、解封和删除用户。
- 审核文章和评论。
- 删除不适当评论。
- 管理分类与标签。
- 查看 AI 使用日志。
- 使用普通用户共有的搜索、资料卡和相册功能。

限制：

- 不能封禁或删除当前登录的管理员账号。
- 不能越权修改其他用户的个人相册。
- 删除已有文章或评论的用户前，需要先处理关联内容。

## 5. 非功能需求

### 5.1 性能需求

- 普通列表和详情接口在本地环境下应快速响应。
- 文章、评论和通知查询应使用索引字段。
- 前端图片使用懒加载，减少初始页面资源压力。
- 私信会话单次最多返回最近 200 条消息。

### 5.2 安全需求

- 所有写操作必须进行登录和权限校验。
- 管理接口仅管理员可访问。
- AI 接口仅登录用户可调用。
- 用户密码不得出现在接口响应中。
- 用户邮箱按访问权限展示：本人及管理员查看完整邮箱，其他普通用户查看脱敏邮箱。
- 前端动态内容进行 HTML 转义，降低脚本注入风险。
- 用户名在业务层和数据库层保持唯一。

### 5.3 可靠性需求

- 已发布内容等业务数据保存到 MySQL；未发布的本地草稿保存在当前浏览器并按账号隔离，刷新页面后不丢失。
- 文章删除使用事务完成级联清理。
- 点赞、关注等关系使用联合主键避免重复。
- 测试脚本执行结束后自动清理临时数据。

### 5.4 易用性需求

- 页面提供明确的成功、失败和加载状态提示。
- 用户名可点击进入资料卡。
- 工作台提供搜索、筛选、分页和索引导航。
- 相册明确显示当前数量和 8 张上限。
- 文章详情提供目录快速定位。

### 5.5 兼容性需求

- 支持 Chrome、Edge 等现代浏览器。
- 支持 Node.js 18+、MySQL 8、Maven 3.8+。
- 后端源码兼容 Java 8，当前环境可使用 JDK 17 运行。

### 5.6 可维护性需求

- 后端采用 Controller、Service、Repository 分层。
- 数据库脚本和运行时迁移逻辑保持同步。
- 前端页面、样式和业务脚本分离。
- README、测试脚本和业务代码同步更新。

## 7. 数据流程说明

### 7.1 顶层数据流

```text
用户/管理员
    |
    | HTTP 请求、表单数据、令牌
    v
Hexo 前端页面
    |
    | REST/JSON
    v
Spring Boot Controller
    |
    v
Service 业务规则与权限校验
    |
    +------> AI 服务/本地审核规则
    |
    v
Repository 数据访问
    |
    v
MySQL 数据库
```

### 7.2 文章数据流

1. 用户在前端输入文章数据。
2. 前端将标题、摘要、正文、分类、标签和附件转换为 JSON。
3. Controller 校验登录状态。
4. Service 校验文章所有权、分类标签有效性并执行 AI 初审。
5. Repository 写入 `articles`、`article_tags`、`article_attachments`。
6. 公开后向关注者写入 `notifications`。
7. 前端重新查询文章流并渲染。

### 7.3 互动数据流

1. 点赞写入 `article_likes` 或 `comment_likes`。
2. 评论写入 `comments`，风险内容标记为 `PENDING`。
3. 内容公开后，系统根据文章作者、父评论作者和操作人生成通知。
4. 用户读取通知时更新 `notifications.read_flag`。

### 7.4 私信数据流

1. 发送方提交消息。
2. Service 判断双方是否互关，并检查非互关发送条数。
3. 合法消息写入 `private_messages`。
4. 系统为接收方写入私信通知。
5. 接收方打开会话后，消息被标记为已读。

### 7.5 相册数据流

1. 新用户创建后写入 8 条默认 `gallery_photos`。
2. `user_gallery_settings` 记录账号已完成默认初始化，防止删除后被重建。
3. 用户上传时先检查当前数量是否达到 8。
4. 用户资料卡按目标用户 ID 查询相册。

## 8. 数据库初步设计

### 8.1 数据表

| 表名 | 主要用途 | 关键字段 |
|---|---|---|
| `users` | 用户和角色 | id, username, password, email, role, banned |
| `categories` | 文章分类 | id, name, description |
| `tags` | 文章标签 | id, name |
| `articles` | 文章主体 | id, author_id, category_id, title, content, status |
| `article_attachments` | 文章附件 | id, article_id, name, data_url |
| `article_tags` | 文章标签多对多关系 | article_id, tag_id |
| `article_likes` | 文章点赞 | article_id, user_id |
| `comments` | 评论与回复 | id, article_id, user_id, parent_id, status |
| `comment_likes` | 评论点赞 | comment_id, user_id |
| `user_follows` | 用户关注关系 | follower_id, followed_id |
| `private_messages` | 私信 | id, sender_id, receiver_id, content, read_flag |
| `gallery_photos` | 用户相册 | id, owner_id, title, image_data_url |
| `user_gallery_settings` | 相册初始化状态 | user_id, initialized_at |
| `notifications` | 通知 | id, user_id, actor_user_id, article_id, type, read_flag |
| `ai_usage_logs` | AI 使用记录 | id, user_id, feature, prompt, thinking, result |

### 8.2 主要关系

- 一个用户可以发布多篇文章：`users 1 : N articles`。
- 一个分类可以包含多篇文章：`categories 1 : N articles`。
- 文章和标签是多对多关系，通过 `article_tags` 连接。
- 一篇文章可以有多条评论和多个点赞。
- 评论支持父子关系，`comments.parent_id` 指向父评论。
- 用户关注关系是用户表的自关联多对多。
- 私信通过发送者和接收者关联两个用户。
- 一个用户最多保留 8 条相册图片记录。
- 通知关联接收用户、触发用户和可选文章。

### 8.3 完整性约束

- `users.username` 唯一。
- 点赞和关注关系采用联合主键，避免重复。
- 文章、评论、相册、私信等通过外键关联用户。
- 删除文章时附件、标签关联、点赞和评论级联清理。
- 删除用户时关注、相册和私信等账户数据级联清理。

## 9. AI 功能需求说明

### 9.1 AI 写作辅助

- 根据标题和正文生成 4—6 条文章大纲。
- 根据正文生成适合首页展示的摘要。
- 根据内容推荐 3—6 个标签。
- 优先从已有分类中推荐分类，也允许推荐新分类。

### 9.2 AI 博客问答

- 用户可以对博客内容提出自然语言问题。
- 系统结合已公开文章生成回答。
- 返回可能的依据文章标题。
- AI 接口必须登录后使用。

### 9.3 AI 内容初审

- 文章和评论发布前进行风险识别。
- 重点识别广告引流、辱骂攻击、低俗色情、博彩诈骗等内容。
- 正常内容直接公开。
- 风险内容仅标记为待审核，不由 AI 最终删除。
- 管理员拥有最终通过或驳回权。

### 9.4 AI 日志

- 记录功能名称、提示词、处理摘要、输出结果和时间。
- 管理员可查看日志。
- API Key 通过环境变量配置，不写入仓库。
- 外部模型不可用时应返回明确错误；只有显式开启时才允许本地兜底。
